# 큰 작업 진행 가이드

## 목적

- 큰 기능, 다중 파일 변경, 리팩터링과 장기 작업의 목표와 범위를 명확히 한다.
- 여러 대화나 날짜에 걸쳐 작업해도 결정과 진행 맥락을 잃지 않게 한다.
- 실제 구현 상태와 검증 결과를 `../work_plans/`에서 추적한다.

## 큰 작업 판단 기준

다음 중 하나 이상에 해당하면 작업 계획을 만드는 것을 원칙으로 한다.

1. 세 개 이상의 파일을 수정할 가능성이 높다.
2. Domain, Activity, Runtime, Overlay, Settings, PetLibrary 중 여러 계층을 함께 변경한다.
3. 제품 동작이나 저장 스키마가 달라진다.
4. 기존 동작 유지가 중요한 리팩터링이다.
5. 사용자와 중간 결정 또는 추가 확인이 필요하다.
6. 한 번에 끝내기보다 단계별 구현과 검증이 안전하다.
7. 작업이 여러 대화 또는 여러 날에 걸쳐 이어질 가능성이 있다.

작은 버그 수정, 문구 변경, 명확한 단일 파일 수정은 별도 계획 없이 진행할 수 있다.

## 진행 절차

1. 최상위 `AGENTS.md`와 관련 프로젝트 문서 및 명세를 읽는다.
2. `../work_plans/INDEX.md`에서 같은 목표의 기존 작업이 있는지 확인한다.
3. 기존 계획이 있으면 새 파일을 만들지 않고 해당 계획을 갱신한다.
4. 새 큰 작업이면 `../work_plans/tasks/YYYY-MM-DD-short-task-name.md`를 만든다.
5. 목표, 범위, 제외 범위, 열린 질문, 작업 순서와 검증 방법을 기록한다.
6. 제품 동작을 바꾸는 열린 질문은 구현 전에 사용자와 결정한다.
7. 구현 중 단계가 완료될 때마다 체크리스트와 진행 로그를 갱신한다.
8. 관련된 가장 좁은 테스트부터 실행하고 결과를 기록한다.
9. 완료 시 결과, 남은 위험과 후속 작업을 기록한다.
10. `../work_plans/INDEX.md`의 상태와 날짜를 갱신한다.

## 상태 값

- `planned`: 계획 작성 완료, 구현 전
- `in_progress`: 작업 진행 중
- `blocked`: 사용자 결정이나 외부 조건을 기다리는 중
- `paused`: 의도적으로 보류
- `done`: 구현과 필수 검증 완료

## 파일명 규칙

```text
work_plans/tasks/YYYY-MM-DD-short-task-name.md
```

파일명의 작업명은 소문자 영문과 하이픈을 사용한다.

## 문서 동기화 규칙

- 제품 범위가 바뀌면 `../project/PRODUCT.md`를 갱신한다.
- 장기 개발 순서나 단계 상태가 바뀌면 `../project/ROADMAP.md`를 갱신한다.
- 아키텍처 경계나 저장 방식이 바뀌면 `../project/ARCHITECTURE.md`와 `../project/DECISIONS.md`를 확인한다.
- 행동 또는 패키지 규격이 바뀌면 `../specifications/`의 해당 명세를 갱신한다.
- 작업 진행 세부사항을 프로젝트 문서에 누적하지 않고 해당 작업 계획에 기록한다.
