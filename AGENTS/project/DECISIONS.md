# MonglePet 결정 기록

프로젝트 전체에 영향을 주는 확정된 제품 및 기술 결정을 기록한다. 결정이 바뀌면 기존 항목을 지우지 않고 상태와 대체 결정을 남긴다.

## D-001 독립 코드베이스

- 상태: accepted
- 결정일: 2026-07-21
- 결정: `bssm-oss/desktop-pet`과 유사 제품은 기능 및 UX 참고 자료로만 사용하고 포크하거나 소스 구조를 복사하지 않는다.
- 이유: MonglePet의 제품 규칙, 펫 패키지, 성능 및 개인정보 기준에 맞는 독립적인 구조가 필요하다.

## D-002 macOS 우선 네이티브 구현

- 상태: accepted
- 결정일: 2026-07-21
- 결정: 초기 버전은 macOS 14 이상, Swift 6, AppKit, SwiftUI, Core Animation으로 구현한다.
- 이유: 투명 오버레이, 시스템 이벤트, 저전력 애니메이션을 macOS 기본 기술로 제어하기 적합하다.

## D-003 파일 기반 로컬 저장소

- 상태: accepted
- 결정일: 2026-07-21
- 결정: Xcode 프로젝트 생성 시 Storage는 `None`을 선택하고, 앱 설정은 버전이 지정된 JSON, 펫 라이브러리는 파일 시스템으로 저장한다.
- 이유: MVP 데이터는 관계형 객체 그래프보다 설정 스키마와 패키지 파일에 가깝고 향후 Windows 구현과 스키마를 공유하기 쉽다.

## D-004 개인정보 최소 처리

- 상태: accepted
- 결정일: 2026-07-21
- 결정: 전면 앱 bundle identifier, 입력 후 경과 시간, 화면 잠금 및 절전 상태만 로컬에서 처리한다. 실제 키 입력, 창 제목, 문서명, 브라우저 주소, 화면 내용은 읽거나 저장하지 않는다.
- 이유: 자동 행동에 필요한 정보만 사용하면서 별도 접근성 권한과 개인정보 위험을 최소화한다.

## D-005 데이터 전용 펫 패키지

- 상태: accepted
- 결정일: 2026-07-21
- 결정: `.monglepet` 패키지에는 JSON과 이미지 데이터만 허용하고 실행 코드, 스크립트, 플러그인을 허용하지 않는다.
- 이유: 외부 펫 등록 기능의 보안성과 이식성을 확보한다.

## D-006 Windows 별도 구현

- 상태: accepted
- 결정일: 2026-07-21
- 결정: macOS 버전이 안정화된 뒤 Windows UI, 시스템 감지, 렌더러를 별도 네이티브 앱으로 구현한다. 데이터 스키마와 테스트 시나리오만 공유한다.
- 이유: 플랫폼별 창 관리와 시스템 API 차이를 억지로 공통화하지 않기 위해서다.

## D-007 앱 식별자

- 상태: accepted
- 결정일: 2026-07-21
- 결정: 앱 Bundle Identifier는 `kr.mapleroom.MonglePet`을 사용한다.

## D-008 메뉴 막대 중심 앱

- 상태: accepted
- 결정일: 2026-07-21
- 결정: MonglePet은 Dock 아이콘이 없는 메뉴 막대 중심 앱으로 구성한다. AppKit `NSStatusItem`을 기본 진입점으로 사용하고 설정창은 필요할 때만 연다.
- 이유: 펫 창을 닫거나 숨긴 상태에서도 깨우기, 설정과 앱 종료 기능에 항상 접근할 수 있어야 한다.

## D-009 펫 오버레이 창

- 상태: accepted
- 결정일: 2026-07-21
- 결정: 펫은 투명하고 테두리가 없는 non-activating `NSPanel`에 표시한다. 패널은 일반 앱 위와 모든 Space 및 전체 화면 보조 공간에 표시하되 화면 보호기나 화면 잠금보다 위에 표시하지 않는다.
- 이유: 사용자의 현재 작업을 활성화 해제하지 않으면서 데스크톱 펫을 지속적으로 보여주기 위해서다.

## D-010 1단계 임시 펫 자산

- 상태: accepted
- 결정일: 2026-07-21
- 결정: 1단계 오버레이 검증에는 투명 배경의 교체 가능한 정적 PNG 한 장을 사용한다. 정식 WebP 아틀라스와 모션 재생은 2단계에서 구현한다.
- 이유: 창 생명주기와 상호작용을 애니메이션 및 패키지 로더와 분리해 먼저 검증하기 위해서다.

## D-011 Codex WebP 아틀라스 가져오기

- 상태: accepted
- 결정일: 2026-07-21
- 결정: Codex legacy v1 8×9 WebP 아틀라스와 v2 8×11 WebP 아틀라스를 가져오기 형식으로 지원한다. 가져온 데이터는 MonglePet 내부 manifest로 변환하며 Codex 고정 행 구조를 내부 런타임 모델로 직접 사용하지 않는다.
- 이유: 기존 Codex 펫 자산과의 호환성을 제공하면서 MonglePet의 모션 및 패키지 규격을 독립적으로 유지하기 위해서다.
- 비고: animated WebP 단일 파일 재생은 이 결정의 범위가 아니며 MVP 가져오기 대상에서 제외한다.

## D-012 1단계 표시 상태와 위치 복구

- 상태: accepted
- 결정일: 2026-07-21
- 결정: 1단계의 재우기는 펫 패널을 파기하지 않고 숨기며, 깨우면 실행 중 마지막 위치를 사용한다. 디스플레이 구성이 변경되어 패널이 화면 밖에 있으면 가장 많이 겹치는 사용 가능 화면 안으로 보정하고, 어느 화면과도 겹치지 않으면 주 화면으로 복구한다.
- 이유: 숨긴 펫을 즉시 복구하면서 디스플레이 분리 후 조작할 수 없는 위치에 남는 상황을 방지하기 위해서다. 앱 재실행 후 위치 영구 복원은 5단계 설정 저장 범위에서 다룬다.

## D-013 ZIP 패키지 해제

- 상태: accepted
- 결정일: 2026-07-21
- 결정: ZIPFoundation 0.9.20을 Swift Package로 정확히 고정해 사용한다. 패키지 전체 해제 편의 API에만 의존하지 않고 모든 엔트리의 경로, 종류, 중복, 크기와 압축률을 먼저 검사하며 심볼릭 링크를 거부한 뒤 개별 추출한다.
- 이유: App Sandbox 안에서 외부 프로세스 없이 ZIP을 처리하면서 zip-slip, 심볼릭 링크 탈출과 압축 폭탄을 MonglePet 규칙으로 방어하기 위해서다.
- 비고: ZIPFoundation의 MIT 라이선스 고지는 공개 배포 준비 시 포함한다.

## D-014 호환 입력의 내부 패키지 변환

- 상태: accepted
- 결정일: 2026-07-21
- 결정: Codex v1/v2, GIF, APNG와 PNG 시퀀스는 런타임에서 별도 형식으로 직접 재생하지 않고 임시 스키마 버전 1 `.monglepet` 디렉터리로 변환한 뒤 기존 `PetPackageLoader`로 다시 검증한다.
- 이유: 호환 형식별 파싱을 설치 및 재생 경계와 분리하고, 모든 가져온 펫에 동일한 보안 제한과 런타임 모델을 적용하기 위해서다.
- 비고: Codex 원본 atlas는 변경하지 않고 복사하며, GIF/APNG/PNG 시퀀스는 투명 PNG atlas의 반복 `idle` 모션 하나로 정규화한다.

## D-015 행동 전환 경계와 규칙 순서

- 상태: accepted
- 결정일: 2026-07-22
- 결정: 일반 행동 목록 변경은 현재 `BehaviorStep`의 `duration`이 끝나는 단계 경계에서 적용하고, 일회성 상호작용만 즉시 시작한다. 자동 규칙은 종류별 결정 우선순위를 먼저 적용한 뒤 큰 `priority` 값부터 평가하며 같은 값은 설정 배열 순서를 유지한다.
- 이유: 짧은 입력 변화로 모션이 중간에 끊기거나 반복 전환되는 현상을 막으면서 사용자가 요청한 상호작용에는 즉시 반응하기 위해서다.
- 비고: 10분 이상 유휴 규칙은 장시간 유휴로 분류해 앱 규칙보다 먼저, 10분 미만 유휴 규칙은 앱 규칙 다음에 평가한다.

## D-016 macOS 활동 감지 API와 주기

- 상태: accepted
- 결정일: 2026-07-22
- 결정: 전면 앱은 `NSWorkspace.didActivateApplicationNotification`, 유휴 시간은 `CGEventSource.secondsSinceLastEventType`, 전원·세션 상태는 공개된 `NSWorkspace` 알림으로 감지한다. 유휴 시간은 1초 간격으로만 확인하고 이벤트 상태 변경은 즉시 발행한다.
- 이유: 실제 키 입력이나 화면 내용을 읽지 않고 접근성·화면 기록·입력 모니터링 권한 없이 자동 행동에 필요한 최소 입력을 얻기 위해서다.
- 비고: 비공개 화면 잠금 분산 알림은 사용하지 않는다. 사용자 세션 비활성화 또는 화면 수면을 화면 사용 불가 상태로 보수적으로 합쳐 `ActivitySnapshot.isScreenLocked`로 전달하며, 해당 상태와 시스템 절전 중에는 polling 및 렌더링을 중지한다.

## D-017 설정 저장 검증과 미래 버전 보호

- 상태: accepted
- 결정일: 2026-07-22
- 결정: schema-v1 설정은 Domain과 저장 DTO를 분리하고 저장 시 전체 값을 엄격하게 검증한다. 로드 시에는 잘못된 항목만 독립적으로 복구하며, 손상 파일은 격리하고 현재 앱보다 새로운 스키마 파일은 원본을 유지한 채 설정 쓰기를 차단한다.
- 이유: 일부 사용자 설정 오류 때문에 전체 구성을 잃지 않으면서도, 구버전 앱이 새로운 형식의 파일을 덮어써 데이터가 소실되는 일을 막기 위해서다.
- 비고: 설정 파일은 최대 5MiB, overlay 너비는 96–384pt, 재생 속도는 0.25–4.0, 행동 목록·목록별 단계·자동 규칙은 각각 최대 100개, 시간 값은 최대 24시간으로 제한한다. 로그인 시 실행은 JSON이 아니라 향후 `SMAppService` 상태를 원본으로 사용한다.

## D-018 설정 세션과 overlay 복원

- 상태: accepted
- 결정일: 2026-07-22
- 결정: `AppSettingsSession`을 저장소와 UI·`AppCoordinator` 사이의 단일 실행 상태로 사용한다. overlay는 디스플레이 UUID와 전역 좌표를 저장하고 현재 visible frame으로 보정하며, 드래그 완료 시 저장한다. 크기 조작은 즉시 반영하되 파일 저장은 조작 종료 시 수행한다.
- 이유: 파일 시스템 책임을 SwiftUI와 AppKit에서 분리하면서 연속 슬라이더 입력의 불필요한 디스크 쓰기를 줄이고, 디스플레이 분리 후에도 펫을 조작 가능한 위치로 복구하기 위해서다.
- 비고: 클릭 통과 상태에서도 메뉴 막대와 설정창을 복구 경로로 유지한다. 미래 스키마에서는 영구 편집을 막지만 실행 중 깨우기·재우기는 허용한다.

## D-019 기본 행동 프리셋과 런타임 timer

- 상태: accepted
- 결정일: 2026-07-22
- 결정: 행동 목록이 비어 있는 첫 실행·기존 설정에는 `idle`, `focus`, `rest`, `sleep` 3초 목록과 유휴 2분 `rest`, 10분 `sleep` 규칙을 메모리에서 주입한다. `PetBehaviorRuntime`은 현재 단계의 남은 시간에 맞춘 일회성 timer로 `BehaviorResolver`, `MotionScheduler`와 `FramePlayer`를 연결한다.
- 이유: 기존 schema-v1 파일을 깨지 않고 즉시 사용 가능한 자동·수동 동작을 제공하며, 상시 실행 앱에서 짧은 주기의 별도 polling을 추가하지 않기 위해서다.
- 비고: 기본 프리셋은 다음 사용자 설정 저장 시 함께 기록한다. 임시 단일 이미지는 source rect와 프레임 시간으로 상태를 구분하며, 정식 펫 자산은 같은 motion ID로 교체한다. `playbackSpeed`는 프레임 delay에만 적용한다.

## D-020 행동 루틴 편집과 펫 애니메이션 구분

- 상태: accepted
- 결정일: 2026-07-22
- 결정: 사용자 UI에서는 `BehaviorSequence`를 **행동 루틴**, `PetMotion`을 **펫 애니메이션**으로 구분한다. 행동 루틴은 현재 펫이 가진 애니메이션 단계만 순서대로 조합하며 다른 행동 루틴을 단계로 중첩하지 않는다.
- 이유: 재생 가능한 원본 애니메이션과 사용자가 구성하는 시간 기반 조합을 명확히 구분하고, 중첩 루틴의 순환 참조와 모호한 재생 의미를 방지하기 위해서다.
- 비고: 내장 루틴은 삭제할 수 없지만 편집할 수 있다. custom 루틴 삭제 시 이를 참조하는 자동 규칙을 함께 제거하고 수동 선택을 안전한 기본값으로 복구한다. 현재 펫에 없는 저장된 애니메이션 ID는 설정에서 소실하지 않고 표시하며 runtime에서 `idle` 폴백을 적용한다.
